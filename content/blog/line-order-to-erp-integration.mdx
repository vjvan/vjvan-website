很多老闆以為，導入 LINE 下單就是數位轉型。其實真正的斷點，在後面那一哩：客戶在 LINE 下了單，這筆資料要怎麼自動進到公司原本在用的 ERP，變成可以出貨、對帳、分析的正式單據。這一段沒打通，LINE 就只是另一個更亂的訊息收件匣。

我是允雷，AI 商業系統架構師。這一年我幫好幾家南部的 B2B 企業搭 LINE LIFF 接單系統，最後都會碰到同一個關鍵問題：訂單怎麼接回 ERP。這篇就把這件事講透。

## 先說一個常見的誤會

老闆們常把這兩件事畫上等號：

- 「我有 LINE 官方帳號了，客戶會用 LINE 跟我下單」
- 「所以我已經數位轉型了」

這中間少了一大段。客戶用 LINE 傳「我要鯖魚 5 片、花枝丸 3 包」，這段文字停在員工的手機裡。接下來發生的事，幾乎都還是人工：

| 環節 | 實際在做的事 |
|------|------------|
| 讀單 | 員工一則一則看 LINE 訊息，自己判斷品項和數量 |
| 抄單 | 手動輸入到 Excel 或 ERP，聽錯、打錯時有所聞 |
| 對價格 | 不同客戶不同價，靠記憶或翻舊單 |
| 出貨對帳 | 訂單和 ERP 兩邊各記一次，數字常對不起來 |
| 看數據 | 這個月賣多少、哪個客戶快流失，月底才手算 |

LINE 只解決了「客戶把需求講出來」這一步。後面的讀單、抄單、進 ERP，全部還是人在接力。訊息量一大，反而比電話接單更亂，因為對話散在好幾個群組和私訊裡，沒有一個地方是「正式訂單」。

## 真正的斷點：LINE 和 ERP 中間沒有橋

台灣中小企業的帳務和庫存中樞，多半是國產 ERP，像正航、鼎新、文中這幾套。這些系統把進貨、庫存、應收應付、報表管得很好，是公司的後台骨幹。

問題是，ERP 是設計給「公司內部人員操作」的。它沒辦法讓外面幾十上百家下游客戶，自己打開來下單。於是就出現一個尷尬的空白：

> 前面是客戶習慣用的 LINE，後面是公司在用的 ERP，中間那座橋沒人搭。

所有抄單的人力、所有打錯的單、所有對不起來的帳，都卡在這座橋的缺口上。我做的事，就是把這座橋搭起來。

## 我的解法：LINE LIFF 當 ERP 的「前端收單層」

LIFF 是 LINE 官方的網頁框架，可以讓一個網頁直接長在 LINE 裡面。客戶不用下載任何 App，在 LINE 點一下圖文選單，就進到專屬的訂貨頁面。

我把整個系統的角色定位得很清楚，分三層：

| 層級 | 角色 | 由誰負責 |
|------|------|---------|
| 前端收單層 | 客戶在 LINE 裡瀏覽商品、選規格、下單、查歷史訂單 | LINE LIFF（我做的部分） |
| 中介整合層 | 把訂單轉成 ERP 看得懂的格式，自動拋轉進去 | 整合引擎（我做的部分） |
| 帳務庫存中樞 | 進貨、庫存、出貨、對帳、財報 | 既有 ERP，例如正航（不動它） |

重點在於：我不是要取代 ERP，而是補上它前面缺的那一層。ERP 繼續當後台骨幹，LIFF 負責把外部客戶的訂單，乾淨、結構化地送進去。

## LINE LIFF 怎麼把訂單送進 ERP？三種對接方式

這是企業主最關心，也最容易被話術唬住的地方。我把實務上的三種做法攤開講，包含各自的取捨，不誇大「一鍵直連、什麼系統都能接」。

| 對接方式 | 怎麼運作 | 適合情境 | 取捨 |
|---------|---------|---------|------|
| 批次匯入（Excel 自動轉入） | 系統把訂單整理成 ERP 的標準匯入格式，定時或一鍵匯入 | 相容性最高，幾乎所有 ERP 版本都支援 | 不是即時，但對「每日截單、隔日出貨」的 B2B 節奏完全夠用 |
| WEB API 串接 | 透過 ERP 的 WEB API 把訂單直接拋轉進去。正航這類國產 ERP 的 API 多採 JSON 格式介接，並具加密與驗證機制 | ERP 版本有開放 API、需要較即時的雙向同步 | 需確認原廠版本、授權與端點規格，依官方文件評估 |

以正航為例，官方就把異質系統整合講成「Excel 自動轉入」與「WEB API 串接」雙軌方案，讓企業不必大改現有系統就能對接。新世代版本（像正航近期推出的 N5）延續同一套整合邏輯，差別主要在 API 的即時性與未來擴充性。

我的原則是：先看企業現在用的是哪一套 ERP、哪個版本，再決定走哪條路。多數情況下，批次匯入就能穩定解決問題，不需要為了「即時」這兩個字硬上 API、增加成本和故障點。能用最簡單的方式打通，就不用複雜的。這是對客戶負責，不是對技術炫技。

## 為什麼是互補，不是取代？

我接案有三條清楚的界線，這三件事我不碰：

- 不做會計總帳
- 不做進銷存核心
- 不做生產與人資管理

這些是 ERP 的本業，它做了幾十年，做得比任何新系統都穩。我做的是「客戶的客戶」這一端：外部下游怎麼把訂單交進來。兩者是魚幫水、水幫魚的關係，不是誰取代誰。

對企業主來說，這代表你原本的 ERP 投資完全不浪費，導入風險也最低。你不需要打掉重練，只需要在前面接上一層收單系統。

## 反過來，ERP 也能透過 LINE 主動找客戶

對接這件事是雙向的。除了「LINE 訂單進 ERP」，還有一個方向正在發展：「ERP 的事件，主動推 LINE 通知給客戶」。

舉例來說，ERP 裡有很多「該有人處理了」的事件：安全庫存不足、廠商交期快到了、應收帳款逾期、某個老客戶太久沒回購。這些提醒在 ERP 裡通常只會在員工登入系統時跳出來，或寄一封 email，第一線人員和客戶不一定會即時看到。但如果把這些事件透過一個通知閘道轉成 LINE 訊息，就能在對的時間，自動推給對的人。訂單一成立自動推確認、出貨自動推到貨提醒、客戶太久沒下單自動喚回，這些對我來說都是現成的能力。

這讓 ERP 從一個「員工才看得到的內部系統」，延伸成一個「會主動跟客戶說話」的營運引擎。

## 為什麼我只做企業，而且堅持一個人做？

講點我自己的觀點。

我只服務企業公司，不做個人。因為這套系統的價值，是幫公司把重複的人力解放出來、把散落的資料收攏成資產，這是企業才會真正願意投資的解決方案，個人用不到這個量級。

而且目前每一個案子，都由我一個人從頭做到尾。這不是逞強，是因為這類系統最難的不是寫程式，而是徹底搞懂一家公司的真實流程：他們怎麼接單、怎麼分客戶價格、怎麼出貨、ERP 怎麼跑。只有完整理解整條流程，才設計得出真正合用的系統。交給多人協作或外包，溝通的耗損反而會讓成品走樣。一個人，反而能完整且快速地把它做完。

維護的部分我也想好了。合作會分成系統建置費和月服務費兩塊，系統交付後持續有人顧、持續優化，不是做完就消失。確定合作時我們再細談。

## 結語：讓 LINE 上的資料，變成企業的數位資產

回到最開頭那個誤會。導入 LINE 下單只是起點，不是終點。真正的數位轉型，是讓客戶在 LINE 上的每一次互動，都自動沉澱成公司可以使用的資料：誰買了什麼、多久補一次、哪些品項在成長、哪些客戶在流失。

當這些資料順暢地流進 ERP，再從 ERP 流回客戶手上，LINE 就不再只是一個聊天工具，而是企業最前線的數位資產入口。
